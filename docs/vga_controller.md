# VGA Controller

## Overview

The VGA controller is a custom AXI4-Lite peripheral implemented on the Zynq PL (programmable logic). It drives a 640×480 display at 60 Hz, receiving pixel data written by the ARM processor over the AXI bus and converting it to the analogue VGA signal the monitor expects.

The design is adapted from Pong Chu's *FPGA Prototyping by SystemVerilog Examples*, ported from his custom FPro bus to AXI4-Lite so it integrates directly with the Zynq PS memory map.

---

## Hardware Architecture

### Signal path

```
ARM CPU (PS)
    │  AXI4-Lite writes
    ▼
AXI Slave Wrapper (video_axi_v1_0)
    │  Internal bus (21-bit address, 32-bit data)
    ▼
Video Controller (chu_video_controller)
    │
    ├──► Frame Buffer Core (chu_frame_buffer_core)
    │        │  pixel writes → Block RAM (320 KB)
    │        │  pixel stream → VGA sync module
    │
    └──► Video Slot Array (up to 8 overlay cores)
             │  e.g. bar pattern generator (slot 0)
             ▼
         Daisy-chain blending (video_sys_daisy)
             │
             ▼
         VGA Sync (vga_sync)
             │  hsync, vsync, 12-bit RGB
             ▼
         VGA connector (Zedboard VHDCI → VGA adapter)
```

### Address decoding

The 21-bit internal address space is split by bit 20:

| Bit 20 | Target |
|--------|--------|
| `1` | Frame buffer (pixel RAM) |
| `0` | Video slot registers (bits 16:14 select the slot) |

The ARM writes a pixel by computing `offset = y * 640 + x` and writing to `VIDEO_FRAME_BASE + 4 * offset`. The frame buffer is **write-only** from the CPU side — reads return zero.

## VGA Sync Generation (`vga_sync.sv`)

Standard 640×480 @ 60 Hz timing is hardcoded:

| Parameter | Value |
|-----------|-------|
| Horizontal display | 640 px |
| H front porch | 16 clocks |
| H sync pulse | 96 clocks |
| H back porch | 48 clocks |
| H total | 800 clocks |
| Vertical display | 480 lines |
| V front porch | 10 lines |
| V sync pulse | 2 lines |
| V back porch | 33 lines |
| V total | 525 lines |

A 25 MHz pixel clock (generated by the MMCM from the Zynq 125 MHz PL clock) drives the horizontal and vertical counters. `hsync` and `vsync` are asserted in their respective retrace intervals. RGB output is forced to zero outside the active display area.

A two-state FSM (`frame_sync` / `disp`) ensures the pixel stream stays aligned to frame boundaries, discarding partial frames caused by late or malformed pixel data from the CPU.

## Frame Buffer and Pixel Format

Pixel data is stored in Block RAM on the FPGA fabric. The frame buffer uses a 19-bit address space, split across two BRAM blocks — a 256K-entry block and a 64K-entry block — to cover the 307,200 pixels of a 640×480 frame without inferring a wasteful single 512K block. Total BRAM allocated: 320K entries × 9 bits = **360 KB**.

**Pixel format — 9-bit packed colour, 3 bits per channel:**

```
Bits [8:6] = Red   (3 bits)
Bits [5:3] = Green (3 bits)
Bits [2:0] = Blue  (3 bits)
```

A palette circuit (`frame_palette_9`) expands each 3-bit channel to 4 bits for the 12-bit VGA DAC by replicating the MSB of each channel:

```
r_out[3:0] = {r_in[2:0], r_in[2]}
g_out[3:0] = {g_in[2:0], g_in[2]}
b_out[3:0] = {b_in[2:0], b_in[2]}
```

This maps `0b000 → 0x0` and `0b111 → 0xF`, giving a linear 8-step range per channel.

Example colour values written by the CPU:

| Colour  | 9-bit value | Binary `RRR GGG BBB` |
|---------|-------------|----------------------|
| White   | `0x1FF`     | `111 111 111`        |
| Black   | `0x000`     | `000 000 000`        |
| Red     | `0x1C0`     | `111 000 000`        |
| Green   | `0x038`     | `000 111 000`        |
| Blue    | `0x007`     | `000 000 111`        |
| Yellow  | `0x1F8`     | `111 111 000`        |
| Cyan    | `0x03F`     | `000 111 111`        |
| Magenta | `0x1C7`     | `111 000 111`        |

A bypass register (written via address `0xFFFFF`) allows the frame buffer output to be overridden by the upstream daisy-chain signal, enabling overlay cores to take priority.

---

## Overlay Slots

Up to 8 video overlay cores can be connected in a daisy chain downstream of the frame buffer. Each core receives the upstream pixel stream and can either pass it through or substitute its own pixels (e.g. a colour bar generator for test). Slot 7 in this project is a bar pattern generator used during bring-up.

## Software Interfaces

### C++ Driver (`FrameCore` — `vga_core.cpp`)

Used by the standalone test application (`zed_vga_test`). Wraps the AXI pixel writes in a class.

**Instantiation:**
```cpp
FrameCore frame(VIDEO_FRAME_BASE);           // 0x40400000
GpvCore   bar(get_video_slot_addr(V7_BAR));  // bar pattern overlay, slot 7
```

**Pixel and screen operations:**
```cpp
frame.wr_pix(x, y, COLOR_RED);               // write one pixel
frame.clr_screen(COLOR_BLACK);               // fill entire screen
frame.plot_line(0, 0, 639, 479, COLOR_GREEN); // Bresenham line
```

**Bypass control:**
```cpp
frame.bypass(1);  // pass upstream signal through — hide frame buffer
frame.bypass(0);  // show frame buffer content
bar.bypass(0);    // enable bar pattern overlay
bar.bypass(1);    // disable bar pattern, pass frame buffer through
```

**Colour constants** (defined in `periph_test.cpp`):
```cpp
#define COLOR_BLACK   0x000   // 000 000 000
#define COLOR_WHITE   0x1FF   // 111 111 111
#define COLOR_RED     0x1C0   // 111 000 000
#define COLOR_GREEN   0x038   // 000 111 000
#define COLOR_BLUE    0x007   // 000 000 111
#define COLOR_YELLOW  0x1F8   // 111 111 000
#define COLOR_CYAN    0x03F   // 000 111 111
#define COLOR_MAGENTA 0x1C7   // 111 000 111
```

### C Driver (`vga_core.c`) — OS integration

Used by the OS application (`zed_os_fpga_app`). Provides a text console API over the same framebuffer hardware, with the same underlying AXI pixel writes.

**Initialisation:**
```c
fbuf_init();   // clears screen, zeroes shadow buffers, sets color = WHITE
```

**Text output:**
```c
color = RED;
kprintf("hello %d\n", 42);  // formatted print to VGA console
kprints("some string");
kputc('A');
```

The global `color` variable is set before any print call. All characters drawn until it is changed will use that colour, allowing per-message coloured output.

**Pixel-level access:**
```c
color = BLUE;
setpix(x, y);   // write one pixel in current colour
clrpix(x, y);   // write black
```

**Colour constants** (defined in `vga_core.h`):
```c
#define BLUE   0
#define GREEN  1
#define RED    2
#define CYAN   3
#define YELLOW 4
#define PURPLE 5
#define WHITE  6
```

These index into an internal lookup table mapping each constant to its 9-bit hardware value.

**Shadow buffers:**

Because the AXI frame buffer is write-only, the C driver maintains two parallel shadow arrays in RAM:

```c
static char cbuf[30][80];    // character glyph at each cell
static int  colbuf[30][80];  // colour index at each cell
```

Every character write updates both the hardware and the corresponding shadow cell. When text reaches the bottom of the 30-row screen, `scroll()` shifts both shadow arrays up one row in software, clears the bottom row, then redraws every cell from the shadow — never reading from hardware. Without this, scrolling would blank the screen because all framebuffer reads return zero.
